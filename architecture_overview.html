1. 專案核心概念

目前 repo 的真正核心不是「角色跑圖」或「戰鬥渲染」，而是把策略、人性、事件與世界壓力組成一個可重現的 simulation loop。 這也是為什麼 SDD 與研發日誌都把 simulation -> CSV -> analysis 當作主契約。

為什麼屬於 simulation system

  • 回合行為、reward、state change、provenance 都是可觀測狀態。
  • 回歸驗證依賴 seed、CSV、analysis，不依賴畫面。
  • 動力學物件是 engine、event、personality、memory，而不是 sprite 或 camera。

為什麼 UI 只是 state visualization

  • UI 應只讀 output_state，不能成為真相來源。
  • GUI 可以被 Web、Godot、CLI replay viewer 取代而不破壞核心。
  • 分析層本身已經是讀 CSV 的獨立驗證面。

為什麼先做文字版 MVP

  • 文字輸出直接對應 state 與 event schema，最容易排錯。
  • 目前已有 seed / CSV / metric 的研究型回歸線。
  • 在核心未穩定前，先讓 transition 可讀、可比較，比做美術更重要。
關鍵原則：engine 應獨立於 GUI。任何畫面、動畫、場景切換，都不能反向定義 state transition。

三層責任

Simulation Core

  • 負責回合推進、reward 計算、state transition、權重更新。
  • 現有關鍵模組：core/game_engine.py、dungeon/dungeon_ai.py、players/base_player.py、evolution/replicator_dynamics.py、simulation/run_simulation.py。
  • 核心規則必須穩定，不可被 UI 或分析反寫。

Content Layer

  • 放置人格定義、事件模板、投影規則、經濟規則、Little Dragon world pressure。
  • 現有錨點是 docs/personality_dungeon_v1/。
  • 它提供的是 schema 與內容，不是 GUI 邏輯。

Interface Layer

  • CLI 是 MVP 與回歸主戰場。
  • Web GUI 是資料檢視與團隊協作層。
  • Godot 是 presentation layer，不應承擔 transition logic。

3. Python Core Engine 架構

這個 repo 的核心程式循環可以壓縮成一句話：input → simulation → output_state。 其中 simulation 是唯一真正有權修改狀態的地方；UI 與 analysis 都不能反向寫入 transition logic。 

flowchart TB
  subgraph PythonEngine[Python Engine]
    I[input]
    S[simulation]
    O[output_state]
    I --> S --> O
  end

  API[API / CSV / JSON Boundary]
  UI[Godot / Web UI]

  PythonEngine <--> API
  API <--> UI

目前 engine 的實際工作順序

  1. BasePlayer.choose_strategy() 依策略權重抽樣。
  2. DungeonAI.resolve_player_outcome() 或 evaluate() 計算 reward。
  3. player.update_utility() 更新累積 utility 與 last_reward。
  4. DungeonAI.update_popularity() 或 expected 模式回寫 popularity。
  5. evolution/replicator_dynamics.py 產生下一輪權重。
  6. simulation/run_simulation.py 寫出 round-level row 與 event JSON。

為何 GUI 不應包含邏輯

  • 如果邏輯寫在 GUI，回歸測試會綁死在畫面框架。
  • UI 變更會污染研究結果，破壞可比較性。
  • CLI、Web、Godot 必須共享同一套 state contract。

為何 engine 不應依賴畫面

  • engine 要能在無頭環境、批次 sweep、測試 runner 中運作。
  • rendering 不應成為 state transition 的前提。
  • 這樣前端才能自由替換。

為什麼 state schema 與 event schema 很重要

State schema

state schema 是 engine 和 GUI、analysis、replay viewer 的共同語言。 它決定 round row 裡什麼叫 p_*、什麼叫 w_*、什麼叫 reward、什麼叫 world reaction。

Event schema

event schema 決定事件如何從內容層進入 runtime，並透過 reward / risk / traits / state effects 改變系統。 它是 content layer 與 simulation core 的主要接口。

simulation cycle

sequenceDiagram

4. GUI 發展路線

依目前架構，GUI 不應被當成核心，而應被視為 presentation layer。 最穩妥的路徑是先用 CLI 鎖定 simulation contract，再走 Web GUI，最後才進入 Godot 前端。 

flowchart LR
  A[CLI MVP] --> B[Web GUI]
  B --> C[Godot Frontend]

  A --- A1[驗證 state / CSV / seed]
  B --- B1[團隊協作 / replay / dashboard]
      C --- C1[presentation / animation / spatial UX]

CLI MVP

  • 用途：鎖定 state contract 與回歸測試。
  • 價值：可批次重跑、可比較、可 debug。
  • 現況：這正是 simulation/run_simulation.py 的主定位。

Web GUI

  • 用途：資料看板、事件 replay、隊友協作、參數調整。
  • 價值：最快把 state 可視化，且便於多人共同編輯與 review。
  • 適合先做，因為它仍然是資料驅動而非引擎綁死。

Godot Frontend

  • 用途：更好的 presentation、動畫、鏡頭、場景感。
  • 價值：讓 narrative / world pressure 有更強的視覺表達。
  • 前提：邏輯必須保持在 Python engine，Godot 只負責 render。
結論：simulation 很適合先用 web UI，因為 web 可以直接讀現有 CSV / JSON / replay 資料；Godot 更適合成為 presentation layer，而不是把邏輯寫死在場景樹裡。

5. Story / Narrative 系統

本專案的故事不應來自固定劇本，而應來自 personality evolution、memory、event chain 與 dungeon response 的疊加。 也就是說，故事是 simulation 之後的解讀結果，而不是一開始先寫死的關卡文案。 

flowchart LR
  E[Layer 1: Event] --> S[Layer 2: State Change]
  S --> M[Layer 3: Memory]
      M --> N[Layer 4: Narrative Interpretation]

Layer 1: Event

  • 事件提供衝突、選項、風險與回饋。
  • 目前事件素材由 docs/personality_dungeon_v1/02_event_templates_v1.json 承載。

Layer 2: State Change

  • 事件的結果會改變 risk、stress、noise、health、intel、popularity 與權重。
  • 這些變化才是 narrative 形成的真正材料。

Layer 3: Memory

  • 短期 memory 由 rolling history、event provenance、threshold state、risk smoothing 等構成。
  • 目前不是一個獨立的「故事資料庫」，而是分散在 runtime state 與輸出紀錄中。

Layer 4: Narrative Interpretation

  • 由分析層或 UI 層把 state trace 轉譯成可讀故事。
  • 適合做 replay viewer、週報、簡報、以及專題敘事。

什麼是 emergent narrative

emergent narrative 的意思是：故事不是先寫好章節，再讓角色去演；而是角色的 personality drift、事件選擇、世界壓力與 memory 累積之後， 從軌跡中「長出來」的敘事模式。

在這個專案裡，analysis/event_provenance_summary.py、seed summary、world update TSV 與 timeseries CSV 都可以成為 narrative 的原料。 這樣寫出來的故事才是從系統內部狀態自然湧現，而不是外掛式編劇。

重點：故事應來自 personality evolution、memory、event chain、dungeon response，而不是固定劇本。劇本如果存在，應該只是一層解釋，不是核心控制器。

6. 組員分工架構

從目前 block workplan 與現有模組邊界來看，最適合分工的是內容、UI、分析、敘事與平行實驗； 最需要由主架構師鎖住的是人格系統、state schema、transition logic 與 engine API。

適合拆給組員的工作

任務 原因
event design 可依現有 schema 增加事件模板、action 權重、failure outcomes 與 text 文案。
memory fragments 可做 short-term memory 文案、回想片段、事件摘要，不必動核心 transition。
lore / narrative text 故事文本屬於解讀層，不應直接寫入 engine 內核。
UI / replay viewer 前端可快速迭代，不需要改動 simulation core。
balancing / analytics 透過 CSV、TSV、summary JSON 做參數調整與回歸分析最有效率。
visual reports 圖表、報告、簡報是對外溝通層，與 engine 分離最安全。

主架構師必須控制的核心

核心 為什麼要鎖住
personality system 人格維度與投影方式是整個 simulation 的語義基礎，不能任意改名或改義。
state schema state schema 是 engine、UI、analysis 的共同契約，破壞後回歸線會失真。
transition logic 回合順序、reward 計算、popularity 更新、event outcome 轉譯都屬於核心動力學。

對應 block workplan：Block 1 應由主架構師把住核心 loop；Block 2 / 3 適合實驗與分析；Block 4 是機制設計；Block 5 則是敘事與文件。

7. Event Schema 設計規範

這個專案的 event schema 已經不是抽象提案，而是可被 EventLoader 驗證與執行的資料契約。 所以設計上要保守：保留既有命名，優先補充內容與 effects，不要任意重塑 runtime 欄位。

概念欄位與現有 runtime 欄位對照

文件中建議的概念 目前 runtime 對應 說明
id event_id 唯一且穩定，應可作為 provenance 的主鍵。
trigger type + risk / success / world gates 表示這個事件屬於哪一類、何時容易被世界抽到、何時容易成功或失敗。
text description 可直接供 UI、replay viewer、報告使用。
choices actions[] 每個 action 會有 name、weights、base_risk、risk_model、success_model 等欄位。
effects reward_effects + state_effects + failure_outcomes 分成 reward、state、failure branch 三段，避免 effect 混雜。

設計規範

  • 事件的 actions[].weights 必須與 dimensions_order 一致，現在是 12 維人格輸入。
  • reward_effects 至少應包含 utility_delta、risk_delta、popularity_shift、trait_deltas。
  • state_effects 的 on_success / on_failure 應只改 stress、noise、risk_drift、health、intel 這類既有 state 變數。
  • failure_outcomes 應明確寫 branch 結果，避免失敗結果由 GUI 臨時推定。
  • EventLoader 目前有 success model registry，包含 linear_risk_complement、logistic、linear_clip、threshold，不要在 schema 中自行發明新的隱性模型名稱。

effect system、personality change、memory change、world reaction

Effect system

  • reward_effects：影響 utility、risk、strategy bias 與 trait drift。
  • state_effects：影響玩家當前狀態與後續 round 的壓力。
  • failure_outcomes：定義失敗分支的代價與次級狀態。

人格、記憶、世界反應

  • personality change 應透過 trait_deltas 進入 player personality。
  • memory change 應透過 rolling history、provenance、threshold state、risk smoothing 保存。
  • world reaction 應透過 event type multipliers 與 Little Dragon 的 pressure feedback 作用。
規範重點：這裡的設計建議不是要改掉既有 schema，而是把現有 schema 的角色說清楚。runtime 仍應保持現有欄位名與驗證邏輯。

8. 未來擴充方向

只要核心是 simulation engine，前端就可以替換。這是本專案最大的工程優勢：未來不是只能長成某一個 UI，而是可以依不同用途長成不同介面。 

flowchart TB
  CORE[Simulation Core]
  WEB[Web Version]
  GAME[2D Game]
  MMO[MMO Simulation]
  AI[AI Training Environment]
  REPLAY[Replay System]
  ANALYTICS[Analytics Platform]

  CORE --> WEB
  CORE --> GAME
  CORE --> MMO
  CORE --> AI
  CORE --> REPLAY
  CORE --> ANALYTICS

Web version

最適合先做。可以直接重用 JSON / CSV / provenance，做事件看板、回放與參數調整。

2D game

適合把 narrative、地城壓力與選項呈現成更完整的遊戲體驗，但不應把動力學塞進場景。

MMO simulation

若未來要擴張，只需要把 core 當作大型世界壓力引擎，而不是把前端當核心。

AI training environment

將 engine 暴露成環境介面，讓 AI 學習 personality evolution 與 event decision making。

Replay system

利用現有 CSV、event JSON、world update TSV 與 provenance JSON 就可以重建歷程。

Analytics platform

analysis 會自然演化成 dashboard / report layer，但不應反向依賴 simulation runtime 物件。

這個架構的核心優勢：因為真實核心是 simulation engine，所以前端可以自由替換。這意味著同一套 core 可以同時服務研究、專題、展示與後續產品化。

10. 最後總結

本專案真正的核心不是 graphics、action combat 或 rendering，而是 state evolution、AI simulation、personality emergence 與 event-driven narrative。

換句話說，遊戲只是 simulation 的其中一種介面。只要 simulation core 穩定，前端就可以自由替換； 只要 state / event schema 穩定，內容就可以持續擴充；只要 engine 與 GUI 解耦，這個專案就可以同時支援研究、展示與後續產品化。

一句話版結論：Personality Dungeon 是一個以人格演化與事件驅動為核心的 simulation runtime，而不是傳統 RPG。RPG 只是它的其中一個外殼。
