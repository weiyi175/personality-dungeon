W3 實驗藍圖

主線 80% 探索 20% Hypothesis Backlog

W3 核心目標：在不擴張機制實作前提下，完成 A3 release 前 gate，並產出可證偽假說 backlog v1，作為 W4 go/no-go memo 的輸入。

1. W3 目標與交付物

工作流 目標 W3 交付物
A3 例行回歸 + release 前 gate 確認主線可安全延續到 W4 A3 release gate note v1
可證偽假說草案（不實作） 把 Event line 重啟條件轉成可驗證假說卡 hypothesis backlog v1
W3 週結論 給出 W4 決策輸入（繼續主線 / 假說優先序） W3 handoff note v1

2. 範圍定義（In / Out）

2.1 In Scope

  • 至少 1 次 A3 例行回歸，並產生 release gate 判定。
  • 整理 W1/W2 失敗樣態證據，產出假說 backlog。
  • 為每個假說寫明可證偽條件與停損條件。

2.2 Out of Scope

  • 不新增機制掃參命令。
  • 不改 runtime 行為契約。
  • 不執行 Event line 重啟實驗（僅規劃假說）。

3. 輸入資料

  1. outputs/track_a_protocol_regression_summary.json
  2. outputs/w2/block_risk_report_v1.json
  3. outputs/w2/block_risk_report_v1.md
  4. outputs/w1/failure_catalog_v1.csv
  5. outputs/w1/d5_w1_review_summary_v1.json

4. 執行流程

4.1 主線 80%：A3 例行回歸 + release gate

./venv/bin/python -m simulation.track_a_protocol_regression

release gate 需明確寫出：

  • overall_pass 是否為 true
  • 核心門檻：L1 <= 3、Healthy >= 42
  • 是否出現主線漂移（相對 W2 最後一次記錄）

4.2 探索 20%：可證偽假說 backlog（不實作）

每張假說卡至少包含以下欄位：

欄位 說明
hypothesis_id唯一識別碼（H-01, H-02...）
target_failure_pattern對應失敗樣態（例：new_l1 cluster）
expected_signal期望改善訊號（例：new_l1 下降、pass_rate 提升）
falsification_rule可證偽條件（明確 fail 判準）
evidence_needed需要哪些既有輸出或新檢查
cost_band估計成本（S/M/L）
risk_note對主線風險與停損點

4.3 假說優先序規則

優先序建議以「資訊增益優先、主線風險次之、成本最後」排序：

  1. 資訊增益高（能直接判別是否值得重啟 Event line）
  2. 主線風險低（不影響 A3 穩定性）
  3. 執行成本可控（優先 S/M）

4.4 H-01~H-03 固定 backlog（本輪鎖定）

W3/W4 期間探索 20% 僅允許以下三張假說卡，不新增機制分支：

Hypothesis 目標 通過條件（摘要） Fail-fast（摘要）
H-01 釐清 broke_non_l1 ~ s3=0.55 閾值群聚是否跨 block 可重現 broke_non_l1 下降且 new_l1=0 任一 block 出現 new_l1 > 0 或 early-trap share 過高即 fail
H-02 優先補齊 cross-block 穩健性證據，不開新機制分支 pass_rate 達標且 mean_new_l1 下降至門檻內 任一 block 出現 new_l1 > 0 或整體通過率不足即 fail
H-03 要求 rescued 與 broke/new_l1 共同改善，避免假陽性 rescued 維持且 (broke+new_l1) 同步下降 只出現救援提升但風險未降，或 A3 drift guard 破壞即 fail

一鍵評估入口（固定）：

./venv/bin/python scripts/w3_hypothesis_eval_v1.py
  • 輸出：outputs/w3/hypothesis_eval_v1.json
  • 輸出：outputs/w3/hypothesis_eval_v1.md

5. W3 時程（五天版）

Day 主線 80% 探索 20% 當日輸出
D1 執行 A3 + 更新 release gate 快照 整理 W1/W2 風險證據 W3 source manifest
D2 主線指標 drift 快檢 產出假說卡初稿 hypothesis draft v0
D3 （如有變更）加跑 A3 補齊可證偽規則與停損條件 hypothesis draft v1
D4 release gate 判讀草案 完成優先序排序 backlog ranking v1
D5 W3 主線結論 W4 handoff 整理 hypothesis backlog v1 + handoff note

6. 驗收標準（Definition of Done）

  1. W3 至少完成 1 次 A3 例行回歸，且 summary 存檔。
  2. release gate note v1 有明確 pass/fail 與理由。
  3. hypothesis backlog v1 至少含 3 張可證偽假說卡，且欄位完整。
  4. W3 handoff note 明確寫出 W4 的 go/no-go 輸入條件。
  5. 全週維持「不做機制擴張掃參」紀律。

8. W3 決策輸入映射

  1. 主線節奏：若 A3 穩定，維持 keep_80_20。
  2. Event line：維持「不重啟」直到出現跨 block 穩健證據且 new_l1=0。
  3. W4 輸入：使用 hypothesis backlog v1 的優先序與可證偽條件。
W3 的探索任務是「寫出可被否證的假說」，不是「先做實作」。
