一、什麼是 SDD？

SDD = Spec-Driven Development（規格驅動開發）

核心思想：

不是先寫程式，而是先寫「不可模糊的規格」。

開發順序變成：

Spec（規格）
↓
Test（驗證）
↓
Implementation（實作）

而不是：

想到 → 寫程式 → 修Bug
二、SDD 包含哪些內容？

你說的「目標、實作方法、步驟、驗收標準」只是其中一部分。

完整 SDD 通常包含：

1️⃣ 目標（Goal）

系統要解決什麼問題？

例如你的個性地下城：

模擬個性如何影響生存策略

讓 Agent 可跨輪次演化

風險不是血量，而是機率崩潰

2️⃣ 規格（Specification）

這是最核心部分。

規格不是描述故事，而是：

狀態定義

變數定義

數學關係

輸入輸出格式

約束條件

例如：

風險值 R ∈ [0, 1]
若 R >= θ → 命運崩潰
成功機率 P_success =
σ(a1*謹慎 + a2*多疑 + a3*好奇)

這些才是 Spec。

3️⃣ 不變條件（Invariant）

哪些事情永遠成立？

例如：

成功率 + 失敗率 = 1

風險值不可小於 0

個性向量總和必須歸一化

這可以防止系統混亂。

4️⃣ 驗證標準（Validation Criteria）

如何知道系統是對的？

不是「看起來合理」，而是：

10000 次模擬後死亡率是否在預期範圍？

不同個性分布是否產生明顯策略差異？

演化後是否降低崩潰機率？

5️⃣ 邊界條件（Edge Cases）

例如：

若所有個性值為 0？

若風險值初始為 0.99？

若成功機率 > 1？

SDD 會強制你寫清楚。

6️⃣ 實作步驟（Implementation Plan）

這時才開始：

模組拆分

介面定義

資料格式

API 設計

7️⃣ 測試設計（Test Design）

例如：

單元測試：事件機率計算

模擬測試：1000 次地下城跑完

演化測試：5 代之後平均風險下降？
