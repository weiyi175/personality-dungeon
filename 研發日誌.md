## 1. 數學模型：理論合理性與「程式對應」

### 1.1 狀態與時間索引（避免「看起來對」其實錯位）

本專案的核心狀態（MVP）可視為三策略：

- 策略集合：`aggressive`, `defensive`, `balanced`
- 每輪的族群行為比例（由抽樣結果得到）：
  - $p(t) = (p_A(t), p_D(t), p_B(t))$，其中 $\sum p_i(t)=1$（樣本數 > 0 時）
- 地下城端的 popularity histogram（計分用）：
  - `DungeonAI.popularity` 是「上一輪」玩家選擇統計（count），用來計算 reward

**程式中的時間順序（以 `core/game_engine.py` 的 `GameEngine.step()` 為準）**：

1) 每個玩家用當前權重抽樣策略（`BasePlayer.choose_strategy()`）。
2) 地下城用「當下 popularity（上一輪統計）」對該策略計分（`DungeonAI.evaluate()`）。
3) 玩家更新 `last_reward` 與累積 `utility`。
4) 這一輪所有玩家都走完後，地下城才把本輪 chosen strategies 寫回 popularity（`DungeonAI.update_popularity()`），供下一輪使用。

因此：
- 第 $t$ 輪的 reward 使用的是 $p(t-1)$（更精準說是上一輪的 count histogram 轉成 proportions）。
- 第 $t$ 輪觀測到的 `p_*` 是本輪抽樣結果；但 payoff 的輸入是上一輪 popularity。

這個 1-step lag 對動力系統是「有意義」的：它引入延遲回饋，讓循環/振盪更可能出現（也更容易受噪音影響）。

### 1.2 Payoff 定義

本 repo 有兩種 payoff mode：

#### (A) `count_cycle`（MVP 版）

對策略 $s$：

- 懲罰：$-\gamma\,n_s$（$n_s$ 是上一輪 popularity count）
- 可選 cyclic bonus：$+\epsilon\,(n_{prev(s)} - n_{next(s)})$

直覺：
- 多人選的策略被懲罰（反羊群）。
- cyclic bonus 讓「前一個策略」相對有利、「下一個策略」相對不利，形成類 Rock–Paper–Scissors 的旋轉結構。

#### (B) `matrix_ab`（研究主線）

定義策略順序（固定環狀順序）：

- `strategy_cycle = [aggressive, defensive, balanced]`

令上一輪比例向量：$x = (x_A, x_D, x_B)$，則 payoff：

$$
A = \begin{pmatrix}
0 & a & -b\\
-b & 0 & a\\
a & -b & 0
\end{pmatrix},\quad U = A x
$$

每位玩家若選擇策略 $i$，其 reward：

$$r_i = base\_reward + U_i$$

理論合理性（為何這樣容易出現循環）：
- 上述 $A$ 是「反對稱 + 旋轉」味道的 payoff 結構（類 RPS），在連續時間 replicator dynamics 下常見封閉軌道/中性循環（取決於是否存在守恆量與對稱性）。
- 本 repo 不是連續時間 replicator；而是離散、有限族群抽樣、且 payoff 有 1-step lag。但這些改動通常會把「理想封閉軌」變成：
  - 有限振盪（Level 1/2）
  - 或被噪音擾動成近似循環的時間序列（需要 Stage2 guard 才不會假陽性）

### 1.3 演化更新：離散近似 replicator（exp-weight）

本 repo 的演化更新在 `evolution/replicator_dynamics.py`：

- 先對每個策略 $s$ 聚合本輪玩家的平均 reward：$\bar u_s$
- 全體平均：$\bar u$
- 權重更新：

$$w_s = \exp(k(\bar u_s - \bar u))$$

其中 $k$ = `selection_strength`。

工程合理性：
- 使用 $\exp(\cdot)$ 保證權重始終為正，避免線性更新 $1+k(\Delta u)$ 可能出現負權重。
- 最後把權重 normalize 到 `mean(weight)=1`，避免整體尺度漂移。

動力系統直覺：
- $k$ 越大，對 reward 差異的反應越「硬」，更容易放大振盪；但也可能引入離散 overshoot。
- $k$ 越小，系統更像平滑近似，振盪可能變弱。

## 2. 可實現性檢查（程式層面）

### 2.1 可重現性（seed 設計）

`simulation/run_simulation.py` 對每位玩家建立獨立 RNG stream：

- 若 `--seed S`：第 i 個玩家使用 `random.Random(S + i)`

這使得：
- 同一組參數 + 同 seed，重跑可重現。
- 不同 seed 的差異主要反映有限族群抽樣的隨機性（也是 stability report 的目的）。

### 2.2 計算成本與可擴展性

- 原始 `simulate()` 會建每輪 row dict（長回合可能記憶體膨脹）。
- 已加入 `simulate_series_window()`：只收集 burn-in/tail 視窗內的 `p_*` 或 `w_*` 序列，對 `rounds=80000` 之類長跑更安全。
- 已加入 `simulation/seed_stability.py --jobs` 的多行程掃描。

結論：
- Level 2（振幅 + 主頻）在工程上已可用「掃參數 + 多 seed」進行研究型驗證。
- Level 3（相位方向一致）在目前動態/定義下，從已產生的 sweep 數據看幾乎不會出現（見第 5 節）。

## 3. 資料契約（CSV schema）與視窗語意

### 3.1 `simulation.run_simulation` timeseries CSV 欄位

輸出欄位（固定契約）：

- `round`
- `avg_reward`：該輪族群平均 reward（可能為空字串）
- `avg_utility`：到該輪為止的平均累積 utility
- `p_<strategy>`：該輪抽樣後的策略比例
- `w_<strategy>`：該輪演化更新後得到的「下一輪抽樣權重」

重要語意：同一列 `round=t` 同時包含：
- 本輪抽樣結果 `p_*(t)`
- 本輪 reward 驅動更新後的 `w_*(t+1)`（以「寫入該列」的方式呈現）

這是合理的（作為下一輪 state），但做分析時要記得它們在概念上是不同時間點的量。

### 3.2 burn-in / tail 視窗（analysis 與 simulation 一致）

cycle metrics 的視窗語意由 `analysis/cycle_metrics.py::_slice_series()` 定義：

- 先丟掉前 `burn_in`
- 若有 `tail`：只取最後 `tail` 長度

`simulate_series_window()` 已做同語意的 window 收集，避免雙重切片錯位。

## 4. Cycle Metrics：分級框架與假陽性防護

本專案採用分級：`cycle_level ∈ {0,1,2,3}`，由 `analysis/cycle_metrics.py::classify_cycle_level()` 實作。

### 4.1 Stage 1：振幅（Amplitude screen）

對每個策略序列 $s(t)$，計算 peak-to-peak：

$$A = \max(s) - \min(s)$$

多策略聚合預設為 `any`：任一策略振幅達標即通過。

**關鍵發現（實驗導向）**：
- 若用 `series=w`（權重），在常見參數點（例如 a=1.0, b=1.2, rounds=8000, burn_in=2400, tail=2000）觀測到的最大振幅約 `0.006~0.007`。
- 因此 `amplitude_threshold=0.02` 會導致 Stage1 全面 fail → grid 全 0。

這是「尺度不匹配」而不是模型不存在循環：權重被 normalize（mean=1）且 exp 更新幅度小，天然振幅較小。

（補）已新增 control cache 產生工具：`./venv/bin/python -m analysis.amplitude_control_cache` 可從 control 的 per-seed `max_amp` CSV 統計 `mean/std` 並輸出 `outputs/control_baseline_stats.json`，供 `simulation.rho_curve --amplitude-control-cache` 使用。

### 4.2 Stage 2：主頻（Dominant frequency via autocorr）

使用簡化自相關找最佳 lag，並以 `corr_threshold` 決定是否通過。

**重要防護（已在程式中實作）**：
- `min_overlap=10`：避免極端大 lag 因重疊樣本太少而產生虛高 corr
- `min_cycles=3`（以 `n//3` cap lag）：避免 lag 過大導致「只看得到不到 1~2 個週期」的 overfit
- 常數序列 guard：近似常數直接回傳無主頻

這些 guard 的目的，是把「有限樣本 + 漂移」造成的假週期，壓到不會過 Stage2。

### 4.3 Stage 3：相位方向一致（Phase direction consistency）

目前 `classify_cycle_level()` 的 Stage3 預設使用 turning-based 的 `phase_direction_consistency_turning()`（更接近「局部轉向一致」的操作性定義），並保留 `phase_direction_consistency()`（centroid-based）作為可選替代：

- 取 $(p_A(t), p_D(t))$ 平面（$p_B$ 由 simplex 約束隱含）
- 固定 centroid $c=(1/3,1/3)$
- 計算每一步的旋轉符號：

$$z_t = \mathrm{cross}(x_t-c,\;x_{t+1}-c)$$

- 計分：

$$score = |\mathrm{mean}(\mathrm{sign}(z_t))| \in [0,1]$$

- 旋轉強度（避免微小噪音）：

$$turn\_strength = \mathrm{mean}(|z_t|)$$

通過條件：
- `score >= eta` 且 `turn_strength >= min_turn_strength`

解讀：
- 若方向近似隨機，正負抵銷 → score 會接近 0。
- 若存在穩定單向繞行 → score 接近 1。

備註：檔內另有 `assess_stage3_direction()`（投影到等邊三角形、用連續三點的 local turning），目前不在 Level 判定主流程中使用，可作為替代診斷工具。

補充（降噪選項）：
- analysis 端：`phase_smoothing=3/5/7...` 可對 Stage3 輸入做移動平均降噪（turning-based 時適用）。
- dynamics 端：`--popularity-mode expected` 會用「權重期望分布」更新 popularity（取代 sampled choices），保留 1-step lag 但降低有限族群抽樣噪音。

補充（Stage3 視窗化，用分位數做穩健偵測）：
- `--stage3-window W --stage3-step S --stage3-quantile q`（預設 q=0.75）
  - 直覺：把 tail 切成多個視窗各自算 Stage3，再取分位數（例如 p75）當作「大多數時間都成立」的方向一致性指標。
  - 用途：coarse 可以用 max（別漏），refine/偵測可以用 p75（別被偶發 burst 騙）。

#### 5.3.1 corr_threshold 校準（series=p）

#### 5.3.4 Grid + Refine 自動化（2026-03-04 初始測試，分數門檻選點）

目的：用「連續分數地形圖」先找候選區，再用多 seed 精煉估計穩健性（避免直接對離散的 Level3 做優化）。

新增工具：`simulation/grid_refine.py`（已落實兩段式：coarse 用 max、refine/報告用 quantile-q；預設 q=0.75，可加嚴到 0.85/0.90）

- Stage3 以滑動視窗計算（避免單一長 tail 把「區段性旋轉」稀釋掉）：
  - `--stage3-window`：視窗長度（步數）
  - `--stage3-step`：視窗滑動步長
  - `--stage3-quantile`：視窗分位數 q（例如 0.75/0.85/0.90；q 越高越嚴格）
- 粗掃（coarse）：在 (a,b) 網格上跑少量 seed（通常 1 個），計算連續指標並用 **max over windows** 取候選（別漏）：
  - `mean_rot_score_max`, `mean_rot_turn_strength_max`
  - 並同時輸出 q 版本供 refine 排序/報告：`mean_rot_score_q`, `mean_rot_turn_strength_q`（每列也會帶 `stage3_quantile` 方便回溯）
  - 以及 `mean_max_amp`, `mean_max_corr`（避免只看到「方向」但其實沒週期）
- coarse 以 `--coarse-top-frac` 自動取 `mean_rot_score_max` 前 X% 當候選點（不需要先猜固定門檻）。
- 細掃（refine）：以候選點為中心做局部細網格，多 seed 重跑並用 **quantile-q over windows** 排序（別被偶發 burst 騙）：
  - refine rank 預設排序鍵是 `mean_rot_score_q`，其次 `p_stage3_pass`

## 6. 理論合理性再檢查：為何 Level2 有、Level3 沒有？

### 6.1 Level 2（振幅 + 主頻）為何合理

即使在理想連續模型中有循環，有限族群 + 離散時間會讓你觀測到的行為更像：

- 某些參數區域有可見振幅（Stage1）
- 但週期性需要足夠「重複週期」與足夠「訊號對噪音比」才能穩定通過 Stage2

你用 `corr_threshold` bracket 的方式，實際上就是在找「訊號足夠但不會因噪音過關」的工作點。

### 6.2 Level 3（相位方向一致）為何在目前設定下不合理/不容易

目前 Stage3 是用連續兩點繞 centroid 的 cross-product sign。

這個指標對下列情況非常敏感：

- **有限抽樣噪音**：$p(t)$ 每步都在 simplex 上抖動，抖動會讓 $z_t$ 正負頻繁翻轉。
- **延遲回饋**（用上一輪 popularity）：容易造成「前後震盪」而非單向繞行。
- **Stage3 用 (pA, pD) 的 2D 投影**：如果主要運動方向剛好在另一個投影方向上，會被弱化。

因此你看到 `mean_stage3_score ~ 0.01` 是一致的：它像「符號幾乎平均」而不是「接近 0.5」。

結論不是「模型錯」，而是「在目前定義下要達 Level3 需要更強的單向旋轉結構」：

- 要嘛動態本身更接近連續 replicator 的中性環（降低噪音、增加玩家數、或改更新形式）
- 要嘛 Stage3 改成更容錯的方向估計（例如連續三點 local turning、或對 phase 做平滑/降噪後再看方向）

#### 8.2.1 Runbook 反饋整理（2026-03-08；方向 1：Refinement-first，不改模型/契約）

目標仍維持：**補資料 → 合併 → 出報告**。本段只把實務風險與對策寫清楚，避免每輪迭代靠手動記憶。

**優點（保持不變）**

- 協議一致（protocol lock）+ prefix 分流，讓 refine 前後可量化比較、可回歸。
- 只補關鍵 N（最吵/最敏感），把算力花在 crossing interpolation noise 上。

**常見風險**

- 選點主觀：只「肉眼看 summary」容易選偏。
- `--in` 清單很長：手動替換 refine CSV 容易漏一個 N，或混入不同 TAG/protocol。
- 噪音殘留：即使加密 k-grid，若 seeds/rounds 不夠或 lowband 邊界不乾淨，$R^2$ 可能只小幅上升甚至下降。
- 時間/資源：`JOBS=0`（或太大）可能太慢或 OOM；中斷後若不善用 resume/縮小 TARGETS，迭代會卡住。

**對策（已落地，可直接用）**

- 客觀選點：用 [analysis/select_noisy_players.py](analysis/select_noisy_players.py)
  - `--mode summary`：用 `*_summary.csv` 快速排序（不需額外欄位）
  - `--mode sweeps`：用你實際餵給 scaling/report 的 `--in` sweep CSV 計算「單調性違反/穿越次數」再排序
- 量化驗證 refine 是否變好：用 [analysis/validate_scaling.py](analysis/validate_scaling.py) 比較 `old_prefix` vs `new_prefix`（`r2/k_inf/c` + 各 N 的 `k50` delta）。
- 降低跑崩成本：
  - refinement sweep 用 `--resume`（[scripts/run_refinement_sweep.sh](scripts/run_refinement_sweep.sh) 已內建）

import csv
import math
from dataclasses import dataclass
from pathlib import Path
from typing import Optional

ctrl_path = Path('outputs/analysis/rho_curve/rho_curve_ctrlmean_plus_refine_summary.csv')
base_path = Path('outputs/analysis/rho_curve/rho_curve_merged_with_lowband_clean_plus_refine_summary.csv')

@dataclass
class Row:
  players: int
  k50: Optional[float]
  k90: Optional[float]
  p3_max: Optional[float]
  k_at_max: Optional[float]
  plateau_k_start: Optional[float]
  plateau_k_end: Optional[float]
  k_first_positive: Optional[float]

def _f(x: str) -> Optional[float]:
  x = (x or '').strip()
  if not x:
    return None
  try:
    v = float(x)
  except Exception:
    return None
  if math.isnan(v) or math.isinf(v):
    return None
  return v

def load(p: Path) -> dict[int, Row]:
  with p.open(newline='') as f:
    r = csv.DictReader(f)
    out: dict[int, Row] = {}
    for row in r:
      players = int(float(row['players']))
      out[players] = Row(
        players=players,
        k50=_f(row.get('k50','')),
        k90=_f(row.get('k90','')),

        plateau_k_end=_f(row.get('plateau_k_end','')),
        k_first_positive=_f(row.get('k_first_positive','')),
      )
  return out

def fmt(v: Optional[float], *, digits: int = 4) -> str:
  if v is None:
    return '—'
  return f'{v:.{digits}g}'

def delta(a: Optional[float], b: Optional[float]) -> str:
  # a: ctrlmean, b: baseline
  if a is None or b is None:
    return '—'
  return f'{(a-b):+.3g}'

rows = [header]

#### 8.11.1 (1) 振幅門檻標準化（Amplitude normalization）

**你要解決的問題**：不同 protocol / payoff 尺度下，固定 `--amplitude-threshold` 很容易「看起來全 fail」，其實只是尺度不對。

**核心想法**：把 Stage1 的有效門檻改成：

$$\text{effective\_threshold} = \text{amplitude\_threshold\_factor} \times \text{control\_reference}$$

其中 control reference 由 control baseline cache 提供（同 series / burn_in / tail）。

**工具與產物**

- 一鍵產 cache（推薦）：[simulation/control_amp_baseline.py](simulation/control_amp_baseline.py)
  - 產物：
    - `--out-per-seed`：per-seed CSV（含 `players,max_amp`）
    - `--out-json`：control cache JSON（供 `simulation.rho_curve --amplitude-control-cache` 使用）
- 手動產 cache（進階）：[analysis/amplitude_control_cache.py](analysis/amplitude_control_cache.py)
  - 用你已有的 per-seed CSV（至少要有 `players,max_amp`）做 JSON 統計

**A. 一鍵建立 control cache（推薦）**

```bash
./venv/bin/python -m simulation.control_amp_baseline \
  --series p \
  --rounds 4000 \
  --seeds 0:29 \
  --players-grid 100,300,500,1000 \
  --burn-in-frac 0.25 --tail 600 \
  --selection-strength 0.02 \
  --popularity-mode sampled --evolution-mode sampled --payoff-lag 1 --init-bias 0.0 \
  --out-per-seed outputs/control/control_amp_per_seed.csv \
  --out-json outputs/control/control_baseline_stats.json
```

**參數（control_amp_baseline）**

- `--series {p,w}`：一定要和之後 `simulation.rho_curve --series` 一致
- `--rounds`：控制組跑多久（越長越穩，但越慢）
- `--seeds`：seed spec（例：`0:29`）
- `--players` / `--players-grid`：單一 N 或多 N cache
- `--burn-in` / `--burn-in-frac`：若不給 `--burn-in`，會用 `rounds * burn-in-frac`
- `--tail`：尾端視窗長度；省略代表使用 burn-in 之後的全視窗（實務建議固定一個 tail 以利可比）
- `--ddof {0,1}`：cache 的 std 計算；預設 1（樣本 std）

**B. 產生 sweep 時啟用 normalization**

在 [simulation/rho_curve.py](simulation/rho_curve.py) 開啟以下 flags：

- `--amplitude-threshold <float>`：原本的絕對門檻（仍保留在輸出做 provenance；但 normalization 模式下主要看 factor × control_reference）
- `--amplitude-normalization {none,control_mean,control_std}`
  - `none`：維持舊行為（只用絕對門檻）
  - `control_mean`：用 cache 的 `mean_max_amp`
  - `control_std`：用 cache 的 `std_max_amp`
- `--amplitude-threshold-factor <float>`：預設 `3.0`（保守；越大越嚴）
- `--amplitude-control-cache <path.json>`：推薦使用 cache，且 loader 會強制檢查 `series/burn_in/tail` 一致
- `--amplitude-control-reference <float>`：手動覆蓋 reference（通常只用於 quick test；會繞過 cache）

**sweep CSV 的 provenance 欄位（會被寫進每個 k 點）**

- `amplitude_normalization`, `amplitude_threshold_factor`, `amplitude_control_reference`
- `amplitude_threshold_effective`, `mean_max_amp_norm`
- `amplitude_control_cache`（使用 cache 時會記錄其路徑）

**調參建議**

- 先用 `control_mean + factor=3.0` 起手式（False Positive Guard 較穩），若完全抓不到 crossing 再逐步放寬：
  - 降 `--amplitude-threshold-factor`（例如 3.0 → 2.0 → 1.5）
  - 或改用 `control_std`（若 mean 太小、std 反映噪音地板更合理）
- protocol 一定要固定：`series/burn_in/tail/rounds/seeds` 不一致會讓 reference 失去意義。

#### 8.11.2 (2) Stage2 主頻顯著性（Frequency significance）

**你要解決的問題**：固定 `corr_threshold` 在不同 tail/T 下可比性差；需要可切換的「顯著性」定義，並把方法與參數寫進 provenance。

在 [simulation/rho_curve.py](simulation/rho_curve.py) 你會用到的參數：

- `--stage2-method {autocorr_threshold,fft_power_ratio,permutation_p}`
  - `autocorr_threshold`：baseline；通過條件 `corr >= corr_threshold`
  - `fft_power_ratio`：通過條件 `power_ratio >= power_ratio_kappa`
  - `permutation_p`：通過條件 `p_value <= permutation_alpha`
- `--stage2-prefilter {0,1}`：預設 1（推薦）；先用便宜 autocorr gate 粗篩，再跑較貴方法
- `--corr-threshold <float>`：baseline autocorr 門檻（也會被 prefilter 使用）
- `--power-ratio-kappa <float>`：fft_power_ratio 的門檻（預設 8.0；越小越寬鬆）
- `--permutation-alpha <float>`：permutation 的 alpha（預設 0.05；越小越嚴）
- `--permutation-resamples <int>`：重抽次數（預設 200；越大越準但越慢）
- `--permutation-seed <int|None>`：要可重現就固定

**sweep CSV 的 provenance / 統計欄位（會被寫進每個 k 點）**

- 設定：`stage2_method`, `stage2_prefilter`, `power_ratio_kappa`, `permutation_alpha`, `permutation_resamples`, `permutation_seed`
- 彙總統計：`stage2_statistic_name`, `mean_stage2_statistic`, `mean_stage2_effective_window_n`

**調參建議（實務）**

- sweep 大面積掃描：建議先用 `autocorr_threshold` 或 `fft_power_ratio`；`permutation_p` 只在 critical band 或少數 N/少數 k 使用。
- 若要讓不同 protocol 比較更公平：優先固定 `effective_window_n`（也就是固定 `burn_in/tail`），再談門檻。

#### 8.11.5 (5) k50 不確定性：Bootstrap baseline + Bayesian（Laplace；stdlib-only）

**你要解決的問題**：k50 點估計沒有顯式不確定性；需要低成本 baseline（bootstrap）與可選的貝氏近似（Laplace）。

工具：[analysis/rho_curve_scaling.py](analysis/rho_curve_scaling.py)

**A. 產 summary（可選 bootstrap / bayes）**

```bash
./venv/bin/python -m analysis.rho_curve_scaling \
  --in outputs/sweeps/rho_curve/main.csv \
  --outdir outputs/analysis/rho_curve \
  --prefix rho_curve_run1 \
  --bootstrap-resamples 2000 --bootstrap-seed 123 \
  --bayes-method laplace --bayes-draws 2000 --bayes-seed 123 --bayes-prior-sigma 10.0 --bayes-assume-increasing 1
```

**參數（rho_curve_scaling）**

- `--targets <csv>`：預設 `0.5,0.9`（用於 k50/k90）
- `--bootstrap-resamples <int>`：`0` 表示關閉；建議 500~5000
- `--bootstrap-seed <int|None>`：固定可重現
- `--bayes-method {none,laplace}`：`laplace` 會以 Binomial-logit + Laplace 近似（聚合 counts；stdlib-only）
- `--bayes-draws <int>`：後驗抽樣數（預設 2000；越大越穩但越慢）
- `--bayes-seed <int|None>`：固定可重現
- `--bayes-prior-sigma <float>`：Normal(0, sigma^2) 的弱先驗尺度（預設 10.0）
- `--bayes-assume-increasing {0,1}`：預設 1；丟棄 `beta<=0` 的 draw（強制 $P(L3)$ 隨 k 單調增加）

**產物（provenance）**

- `outputs/analysis/rho_curve/<prefix>_summary.csv`
  - 會含 `k50_boot_*` 與 `k50_bayes_*` 欄位（若有啟用）
  - 會含 Stage2 settings 欄位（從 sweep CSV 抽 first non-empty）
- `outputs/analysis/rho_curve/<prefix>_k50_fit.json`（若 summary 有足夠 non-censored 點可擬合，會產出線性 $k_{50}(N)\approx k_\infty + c/N$ 的擬合結果）
- `outputs/analysis/rho_curve/<prefix>_bayes_fit.json`（只有 `--bayes-method!=none` 才會產）
  - 內容：bayes 設定 + 每個 N 的 k50_bayes 統計（作為研究 provenance artifact）

## 9. Jacobian / 局部線性化：lagged 的 |λ| 與「螺旋」判讀模板（可直接貼）

目的：把「觀測到的旋轉」和「理論上的旋轉/螺旋（線性化）」接起來。

這裡的核心是看 *均勻內點平衡* $p^*=(1/3,1/3,1/3)$ 附近的 Jacobian：

- **ODE 模式**：2D Jacobian 的特徵值是 $\lambda = \alpha \pm i\,\omega$。
  - $\alpha<0$：穩定螺旋（局部衰減振盪）
  - $\alpha\approx 0$：近似中性旋轉（中心）
  - $\alpha>0$：不穩定螺旋（局部放大振盪）

- **lagged 離散模式**：狀態是 $(p_t, p_{t-1})$（以 $(u,v)$ 表示各 2 維）→ 4 維線性系統。
  - 輸出的 `state eigvals` 是 4 個 $\lambda$。
  - 以 **spectral radius** $\rho = \max |\lambda|$ 判斷局部穩定性：
    - $\rho<1$：局部收斂（若 dominant pair 是複數 → 衰減螺旋）
    - $\rho\approx 1$：近似中性（更容易被噪音/非線性推成漂移或近似極限環）
    - $\rho>1$：局部發散（若 dominant pair 是複數 → 發散螺旋；通常會被非線性飽和成有限振盪）

注意：這是 *局部*（在 $p^*$ 附近）的線性化，並不保證全域行為；但它能回答「這套 payoff+更新規則在理論上是否帶有旋轉結構、以及旋轉是否傾向衰減/放大」。

### 10.1 目前 k50(N) 形狀的物理/統計解讀（iter8）

從 iter8 的點估計來看，`k50(N)` 呈現明顯的非單調結構（N=100 的局部高點、N=200–300 的 dip、之後再回升）。這裡建議把它拆成「統計層面」與「物理層面」兩種可能來源，先用 diagnostics 圖把兩者分開。

**統計層面（需要先排除的 artifact）**

- **模型配適訊號偏弱**：目前 `k50 = k_inf + c/N` 的 R² 只有 ~0.144（iter8），表示「1/N 單一修正」對這組點的解釋力有限；因此不宜把 `k_inf` 的數值視為高精度估計，而應視為一個暫時的工作點。
- **crossing estimator 的敏感性**：`k50` 是由離散 k-grid 上的 `P(Level3)` 線性插值求得；當某些 N 的曲線在 crossing 附近非常陡、或局部出現微小非單調，`k50` 會對單點的 seed variance 特別敏感。
- **殘餘粗糙度集中在少數 N**：在 `eps=0.02` 的 sweeps-mode 評分下，粗糙/不單調主要集中在 N=50 與 N=500；N=100/300 雖然有 crossing（cross=1），但加密 band 後 `k50` 仍不變，較像 estimator 已穩定。

**物理層面（若 diagnostics 支持，才進一步主張）**

- **crossover / 多尺度修正**：非單調的 `k50(N)` 更像「主導修正項之外還有次主導項」或「中間尺寸 crossover」；典型形式可能是 $k_{50}(N)=k_\infty + c/N^\beta + d/N^\gamma$（或在不同 N 區段有效 \beta 不同）。
- **near-critical window 的機率效應**：`P(Level3)` 本質上是有限時間窗內的事件機率；中間 N 的 dip 可能對應「噪音下降速度 vs 動力學尺度（例如延遲回饋造成的有效相關長度/時間）」的競爭，而不是單純的 1/N 平滑趨近。

結論（迭代策略）：既然 refinement 已收斂（iter6–iter8 全零差分），下一步應優先轉向「診斷與解釋」：用可重現圖表檢查 non-monotone 結構是否是系統性偏差（residual/heatmap/collapse）還是少數點的粗糙造成。

### Discussion

事件閉環的核心機制在於：failure_threshold hard gate 限制了高風險行動的後果，
health penalty 與 state decay 作為延遲懲罰迴路，防止 risk/stress 永久飽和，

`w_` 序列在 `selection_strength` 較小時（ss < 0.095）振幅過低，需要 fallback 至
`p_`（simplex proportion）序列才能正確偵測 Stage 1。此設計使分類器對不同 ss 範圍
均穩健，不依賴固定的序列選擇。

目前尚未觀察到 Level 3（結構性循環），推測需要更強的非線性耦合
（如 `stress → 下輪 risk_bias` hook）或更大的 `selection_strength` 才能觸發。

高 k 和高 σ 均有**反效果**（score 下降）；payoff 非對稱在 a=0.6 達到 plateau（0.5356）。
**結論：Stage 3 是現有「replicator + 事件閘」架構的結構性不可達目標，非參數可調。
需要 adaptive rule-mutation（dungeon 規則隨策略分佈動態調整）、多輪歷史整合、或子族群不對稱耦合。**

**EMA risk_ma 實驗（2026-03-19）**：為測試「跨回合歷史整合（路徑1）」是否突破天花板，
實作指數移動平均 `risk_ma = α·risk_ma + (1-α)·risk`，加算至 `final_risk`（`--risk-ma-alpha` + `--risk-ma-multiplier`）。
執行 α ∈ {0.80, 0.85, 0.90} × mult ∈ {0.3, 0.5, 0.7}，共 9 條件，seed=42, rounds=8000。
全部 9/9 Level 2，最佳 score=0.5280（α=0.90, mult=0.5），**低於** baseline 0.5335（gap 擴大至 0.022）。
turn_strength 仍恆定（0.000624–0.000644）。**EMA 路徑宣告失效。**
機制解釋：EMA 持續放大歷史高風險 → 系統性偏高 final_risk → 策略分布更趨雜亂而非更有結構，
天花板為架構性（而非資料稀疏），多輪歷史記憶本身亦無法繞過 replicator 更新的混合效應。

**Stress-dependent decay asymmetry 實驗（2026-03-19）**：為測試「非線性延遲強化」是否誘發 delay-induced bifurcation，
實作 `effective_rate = 1 - (1-base_rate)/(1 + c·stress^β)`（`--stress-decay-c` + `--stress-decay-beta`）。
執行 β=2.0, c ∈ {1.0, 1.5, 2.0, 2.5}，seed=42, rounds=8000。
全部 4/4 Level 2。結果：

| c | Stage 3 score | turn_strength | avg_sr（尾部） |
|---|-------------|--------------|-------------|
| 0（baseline）| 0.5335 | 0.000632 | 0.482 |
| 1.0 | 0.5335 | 0.000632 | 0.471 |
| 1.5 | **0.5349** | 0.000629 | 0.465 |
| 2.0 | 0.5337 | 0.000634 | 0.461 |
| 2.5 | 0.5337 | 0.000634 | 0.461 |

**確認 stress 確實在累積**（avg_sr 從 0.482 降至 0.461），但 turning-consistency score 僅微幅上升（最大 +0.0014 at c=1.5），
turn_strength 全程恆定，gap=0.0151。**非線性 decay 延遲路徑亦宣告實質失效**（僅邊際噪音幅度的改善）。
根本原因：stress 累積只影響 final_risk clipping 頻率（gate 統計），並不改變 replicator 更新中策略間的相位關係。
**→ 下一步：Adaptive rule-mutation（外部動態閘值，雙尺度行為）。**

**Adaptive rule-mutation 實驗（2026-03-19）**：實作周期性 ft 自適應調節
`ft_new = ft_base × (1 + s × (p_agg − 1/3))`，每 500 rounds 更新一次（`--adaptive-ft-strength` + `--ft-update-interval`）。
執行 s ∈ {0.20, 0.30, 0.40, 0.50}，seed=42, rounds=8000。

**結果：4/4 條件與 baseline 完全相同（字元級相同，score=0.5335, ts=0.000632, sr=0.471）。**
決定性實驗：s=0.20 vs s=0.50 的 p_aggressive 在 8000/8000 輪完全相同（max diff=0.00e+00）。
ft 更新在 500-round checkpoints 確實有數值變化（ft_new 範圍 0.686–0.720），但 success_count 在任何一輪都沒有不同。

**根本原因（新發現）**：atractor 將 `final_risk` 分布自組織為**雙峰態**（遠低於或遠高於 ft 門檻），
`final_risk ∈ [0.686, 0.720)` 的樣本數為零。Gate 作為硬二元開關，而非平滑梯度——
這**反過來正是 §4.5 Gate-Stabilised Limit Cycle 機制的最強支持證據**：
limit cycle 軌跡主動迴避 gate 轉換區，使 ft 擾動無效。

**三條路徑系統性失效（EMA → nonlinear decay → adaptive ft），結論強化：**
所有嘗試「從內部 state/gate 層面」誘發 Level 3 的路徑均已窮盡。
要突破天花板，需要在**payoff 矩陣層面**（自適應 a/b）或**族群結構層面**（子族群不對稱耦合）引入質性不同的反饋。

**Adaptive payoff matrix 實驗（2026-03-19）**：嘗試在 payoff 矩陣層面引入反饋

**v1（乘法公式）**：`a_new = a_base × (1 − 0.15 × s × (p_agg − 1/3))`，s∈{0.15, 0.25, 0.35}，interval=500。
實際 a 變化量 <0.3%（因 p_agg 鎖死在 0.24–0.28，delta 極小），3/3 條件與 baseline 完全相同（score=0.5335, ts=0.000632）。

**v2（加法+clipping公式）**：`a_new = clamp(a_base + s×(p_agg−target), 0.5, 1.2)`，target=0.30，
s∈{0.3, 0.6, 1.2}，interval∈{300, 500}，共 6 條件。
s=0.6,間隔300 的 a 實際變化 2–7%（如 t=299: a→0.746），屬真實擾動。

| strength | interval | Stage 3 score | turn_strength | sr |
|---------|---------|-------------|--------------|-----|
| 0.3 | 300 | 0.5335 | 0.000632 | 0.471 |
| 0.3 | 500 | 0.5335 | 0.000632 | 0.471 |
| 0.6 | 300 | 0.5335 | 0.000632 | 0.471 |
| 0.6 | 500 | 0.5335 | 0.000632 | 0.471 |
| 1.2 | 300 | 0.5335 | 0.000632 | 0.471 |
| 1.2 | 500 | 0.5335 | 0.000632 | 0.471 |

**結果：全部 9 條件（合計 v1+v2）score=0.5335，ts=0.000632，sr=0.471——與 baseline 字元級相同。**

**根本原因**：payoff 矩陣改變 5–7% 仍被 Gate-Stabilised Limit Cycle attractor 完全吸收。
`turn_strength` 恆定於 0.000632，證明 orbit shape 由事件 gate 機制決定，與 payoff 絕對量無關。
任何「在 replicator 更新層inject 擾動」而不改變 gate 本身拓樸結構的方案均無效。

**五路徑失效彙總（EMA → stress-decay → adaptive ft → adaptive payoff v1 → adaptive payoff v2）：**

| 路徑 | 機制 | 最佳 score | gap | 根本失效原因 |
|-----|------|-----------|-----|------------|
| EMA risk_ma | 線性歷史記憶 | 0.5280 | 0.022 | 全局放大 risk → 更混亂 |
| Nonlinear decay | stress 自減慢 | 0.5349 | 0.015 | 只改 gate 觸發頻率，非相位 |
| Adaptive ft | 動態 gate 閾值 | 0.5335 | 0.015 | 雙峰空帶，band 零樣本 |
| Adaptive payoff v1 | 乘法 a/b 回饋 | 0.5335 | 0.015 | delta 太小（<0.3%） |
| Adaptive payoff v2 | 加法+clip a/b | 0.5335 | 0.015 | gate 拓樸支配 orbit shape |

**結論**：v1 架構天花板為**結構性**。Gate-Stabilised Limit Cycle attractor 的 basin of attraction
涵蓋所有已測試的參數擾動空間。要突破 Level 3，需要改變 **gate 機制本身的拓樸結構**（如
子族群不對稱耦合使不同 sub-group 呈現不同 phase，或完整移除 event gate）。

#### 13C：cross-term coupling（新增 `matrix_cross_coupling`）

**Spec/實作**：為 `matrix_ab` 新增可選 cross-term `c_AD`（預設 0，不影響既有研究管線）：

$$
K(x; c_{AD}) = c_{AD}\,(-x_D,\,-x_A,\,x_A+x_D)
$$

總 payoff：

$$
U = A x + K(x; c_{AD})
$$

#### 13C 的解釋
